# 浏览器自动化控制总体规划设计

## 1. 总体目标

在现有原生 Function Call 基础上，分阶段引入基于 `chrome.debugger` 的浏览器自动化控制能力。

总体目标不是一次性搬完整功能，而是先搭稳定、可验证、可扩展的地基，再逐步接入快照、基础操作、多步骤任务、安全控制、完整验收。

本规划默认：

- 默认声明 `debugger` 权限。
- 浏览器控制默认关闭，必须用户显式开启。
- UI 必须显示“正在调试 / 风险提示”。
- MCP 暂不实现，只预留后续扩展空间。
- 工具注册表仍是唯一 allow-list，模型不能动态执行未知工具。
- 每一阶段完成后必须同步更新 `AGENTS.md`，沉淀新增约束和验证要求。

## 2. 总体架构

### 2.1 核心模块

#### BrowserControlManager

- 负责浏览器控制总入口。
- 管理目标标签页、开启/关闭状态、连接生命周期、串行执行队列、错误归一化。
- 后续所有浏览器控制工具都通过它执行。

#### BrowserDebuggerConnection

- 封装 `chrome.debugger.attach`、`sendCommand`、`detach`、`onDetach`、`onEvent`。
- 负责启用 CDP domain，例如 `Runtime`、`Page`、`DOM`、`Accessibility`。
- 处理 Chrome 拒绝 attach、用户取消调试、tab 关闭、受限页面等场景。

#### BrowserControlSnapshotManager

- 负责读取 Accessibility Tree。
- 生成模型可读的页面结构快照。
- 维护 UID 与 DOM backend node 的映射，为后续点击、填写、键盘操作打基础。

#### BrowserControlToolExecutor

- 将 Function Call 工具调用转发到浏览器控制模块。
- 第一阶段只实现 `take_snapshot`。
- 后续扩展 `click`、`fill`、`press_key`、`wait_for`、`navigate_page` 等工具。

#### BrowserControlPreference/UI

- 聊天偏好与当前会话覆盖项中增加浏览器控制开关。
- 输入区和设置区展示浏览器控制状态、风险提示、启用/关闭入口。

### 2.2 数据流

1. 用户在 Side Panel 显式开启浏览器控制。
2. 前端保存 `browserControlEnabled = true`。
3. 用户发送聊天消息。
4. appStore 根据偏好决定是否暴露 `browser.*` 工具。
5. 模型通过 Function Call 请求 `take_snapshot` 等工具。
6. background 校验工具是否注册、是否启用、浏览器控制是否开启。
7. `BrowserControlManager` 连接目标标签页并执行工具。
8. 工具结果回灌模型，最终回复展示在聊天中。
9. 用户关闭浏览器控制或目标 tab 失效时，background 主动 detach 并清理状态。

## 3. 阶段一：地基搭建

### 目标

建立浏览器控制能力的最小稳定骨架，但只开放 `take_snapshot` 之前的基础设施，不实现页面操作。

### 实现内容

- 在 `manifest.json` 声明 `debugger` 权限。
- 新增浏览器控制开关，默认关闭。
- 新增用户可见风险提示。
- 新增浏览器控制偏好字段：
  - `browserControlEnabled: boolean`
  - 当前会话可覆盖该字段。
- 新增受限 URL 判断：
  - `chrome://`
  - `edge://`
  - `about:`
  - `chrome-extension://`
  - `view-source:`
  - Chrome Web Store
- 新增 `BrowserDebuggerConnection`。
- 新增 `BrowserControlManager` 基础生命周期：
  - 设置目标 tab
  - attach
  - detach
  - 外部断开处理
  - tab 关闭处理
  - 中文错误归一化
- 关闭浏览器控制时必须立即 detach。

### 验收标准

- 扩展构建产物中包含 `debugger` 权限。
- 浏览器控制默认关闭。
- 用户不开启时不会 attach debugger。
- 用户开启后，目标普通网页可 attach。
- 受限页面返回明确中文错误，不 attach。
- 用户关闭开关后立即 detach。
- tab 关闭或用户取消调试时状态被清理。
- `npm run typecheck` 通过。
- 相关单元测试通过。
- `npm run build:extension` 通过。
- `AGENTS.md` 记录 debugger 权限、显式开关、受限页面和 detach 约束。

## 4. 阶段二：页面快照工具接入

### 目标

完成 `browser.take_snapshot` 的 Function Call 端到端能力，让模型可以读取当前页面结构。

### 实现内容

- 工具注册表新增：
  - `id: "browser.take_snapshot"`
  - `name: "take_snapshot"`
  - 参数为无参数对象。
- 新增 `BrowserControlSnapshotManager`：
  - 启用 `DOM`、`Accessibility`。
  - 调用 `Accessibility.getFullAXTree`。
  - 将 AX Tree 格式化为模型可读文本。
  - 为可交互节点生成稳定 UID。
- background 工具执行器识别 `take_snapshot`。
- 只有以下条件都满足时才暴露工具：
  - 工具调用总开关开启。
  - 浏览器控制开关开启。
  - `browser.take_snapshot` 在启用工具列表中。
- 工具结果通过现有 `toolCallRecords` 展示。
- 工具返回内容作为 tool result 回灌模型。
- 添加浏览器控制专用系统提示基础版：
  - 不允许猜 UID。
  - 需要页面状态时先调用 `take_snapshot`。
  - 工具失败时不要编造结果。

### 验收标准

- 用户开启浏览器控制后，模型可调用 `take_snapshot`。
- 快照结果包含页面标题、URL、可访问节点结构或明确空状态说明。
- 未开启浏览器控制时，即使模型返回 `take_snapshot` 也拒绝执行。
- 伪造 runtime `tools` 不能绕过注册表 allow-list。
- attach 失败、快照失败、空页面均返回中文错误。
- 工具调用记录显示运行中、成功或失败状态。
- OpenAI Chat Completions 与 Anthropic Messages 工具调用链路均通过测试。
- `npm run typecheck`、相关 vitest、`npm run build:extension` 通过。
- `AGENTS.md` 记录 `browser.take_snapshot` 暴露条件与快照约束。

## 5. 阶段三：基础页面操作工具

### 目标

在快照和 UID 体系稳定后，接入最小可用的页面操作能力。

### 实现内容

新增工具：

- `browser.click`
  - 参数：`uid`、可选 `includeSnapshot`。
  - 通过 UID 找 DOM backend node。
  - 优先物理鼠标事件，必要时使用受控 JS fallback。
- `browser.fill`
  - 参数：`uid`、`value`、可选 `includeSnapshot`。
  - 支持 `input`、`textarea`、`select`、`contenteditable`。
  - 处理 checkbox/radio/switch。
- `browser.press_key`
  - 参数：`key`、可选 `includeSnapshot`。
  - 仅允许白名单按键和常见组合键。
- `browser.wait_for`
  - 参数：`text[]`、`timeout`。
  - 等待页面可见文本出现。
  - timeout 必须有上限。
- 可选 `includeSnapshot`：
  - 对成功操作自动附带最新快照。
  - 操作失败时不自动附带，提示重新 `take_snapshot`。

### 验收标准

- 模型能完成简单流程：快照 -> 点击输入框 -> 填写文本 -> 按 Enter。
- 非法 UID、过期 UID、缺失参数、错误类型均返回中文错误。
- 页面变化后旧 UID 不能静默误用。
- `fill` 不把中文输入法中间态写入项目设置；页面输入只通过受控工具执行。
- `press_key` 拒绝危险或未知按键。
- `wait_for` 超时后给出中文提示，不阻塞聊天流程。
- 每个工具都有参数校验测试、成功测试、失败测试。
- `npm run typecheck`、相关 vitest、`npm run build:extension` 通过。
- `AGENTS.md` 记录 UID、includeSnapshot、操作失败恢复规则。

## 6. 阶段四：导航与多页面控制

### 目标

支持更完整的浏览器任务，例如打开页面、跳转 URL、刷新、返回、切换受控页面。

### 实现内容

新增工具：

- `browser.navigate_page`
  - 支持 URL 跳转、back、forward、reload。
  - URL 必须合法，禁止受限协议。
- `browser.list_pages`
  - 列出当前受控范围内可控制标签页。
- `browser.select_page`
  - 根据 `list_pages` 的 index 切换受控目标。
- `browser.close_page`
  - 关闭指定 index 页面。
  - 禁止误关非受控范围页面。
- 可选支持 tab group 或内部受控范围标识。
- 导航后清理旧快照 UID。

### 验收标准

- URL 导航成功后旧 UID 失效。
- 受限 URL 导航被拒绝。
- list/select/close 只作用于受控范围。
- 关闭当前受控页后状态正确切换或清空。
- 导航失败不会留下假连接状态。
- 单元测试覆盖导航、受限 URL、多 tab、关闭 tab。
- 如涉及 UI 状态，补充对应组件测试。
- `npm run typecheck`、相关 vitest、`npm run build:extension` 通过。
- `AGENTS.md` 记录导航、受控范围和旧 UID 失效规则。

## 7. 阶段五：安全增强与高风险能力评估

### 目标

在基础自动化稳定后，再评估是否开放高风险工具。

### 候选能力

- `browser.evaluate_script`
  - 默认不开放。
  - 若开放，优先定位为只读检查、数据提取和计算。
  - 禁止直接返回 DOM node。
  - 限制脚本长度、返回大小、执行超时。
- `browser.attach_file`
  - 默认不开放。
  - 必须用户显式选择文件或确认路径。
  - 只允许 native `<input type="file">`。
- `browser.type_text`
  - 用于富文本编辑器和命令面板。
  - 必须要求已有焦点或前一步明确聚焦。
- `browser.handle_dialog`
  - 处理 alert、confirm、prompt、beforeunload。
  - 必须明确 action。

### 验收标准

- 高风险工具默认关闭。
- 每个高风险工具都有单独用户开关或高级设置。
- 参数、超时、返回大小、错误提示都有测试。
- 不泄漏本地路径、敏感页面内容、异常堆栈。
- `AGENTS.md` 记录高风险工具默认关闭和启用条件。

## 8. 阶段六：完整任务闭环与可验收版本

### 目标

形成可交付的浏览器自动化能力，支持真实用户任务的稳定闭环。

### 实现内容

- 完整工具集稳定：
  - 快照
  - 点击
  - 填写
  - 键盘
  - 等待
  - 导航
  - 页面列表与切换
- UI 状态完整：
  - 未开启
  - 正在连接
  - 已连接
  - 受限页面
  - 调试已断开
  - 工具执行中
  - 工具失败
- 工具调用过程可追踪：
  - 参数摘要
  - 执行状态
  - 结果摘要
  - 失败原因
- 错误恢复策略：
  - UID 过期提示重新快照。
  - 页面跳转后自动清理旧快照。
  - debugger 断开后不继续执行动作。
  - 任务失败不留下永久 streaming 占位。
- 补充端到端冒烟：
  - 打开测试页。
  - 快照。
  - 点击按钮。
  - 填写表单。
  - 提交。
  - 等待结果文本。
  - 最终回复用户任务完成。

### 验收标准

- 真实扩展环境中可完成基础网页表单任务。
- 关闭浏览器控制后不再暴露浏览器工具。
- 受限页面、tab 关闭、debugger 断开、模型乱传参数均可控失败。
- OpenAI 与 Anthropic 工具调用都可用。
- 流式模式下工具执行和最终回复状态正确。
- `npm run typecheck` 通过。
- 全部相关 vitest 通过。
- `npm run build:extension` 通过。
- 至少一个 Playwright 或扩展冒烟验证通过。
- `AGENTS.md` 完整记录浏览器控制工程经验。

## 9. 后续扩展方向

### MCP 远程工具调用

- 不属于当前浏览器自动化地基。
- 后续可作为新的工具来源接入统一工具执行器。
- 必须独立设计 MCP 配置、工具列表、远程调用权限和脱敏策略。

### 任务规划器

- 在工具稳定后，可加入多步骤任务规划提示词。
- 不应在地基阶段引入复杂 Agent 框架。

### 操作审计与回放

- 可保存浏览器工具调用轨迹。
- 后续用于调试、导出或用户审计。

### 更强页面理解

- 可结合截图、视觉模型、DOM、Accessibility Tree。
- 必须继续遵守视觉模型能力标识和图片输入限制。

## 10. 分步实施建议

1. 先实现阶段一：权限、开关、风险提示、连接生命周期。
2. 再实现阶段二：`take_snapshot` 端到端。
3. 阶段一和二稳定后，再进入基础操作工具。
4. 每次只推进一个阶段。
5. 每个阶段开始前单独生成更详细的阶段计划文档。
6. 每个阶段完成后必须验证、更新 `AGENTS.md`，再进入下一阶段。
