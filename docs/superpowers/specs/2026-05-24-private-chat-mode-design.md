# 聊天隐私模式设计

## 目标

为当前聊天增加“隐私模式”：用户可以在普通新对话或空白占位对话中进入隐私模式，隐私模式下的消息只保存在内存中，不写入 IndexedDB，也不出现在历史对话列表。用户点击“保存”后，当前隐私对话会写入本地存储、退出隐私模式，并成为普通历史对话。

## 设计边界

- 已存在且包含历史消息的普通会话不显示“隐私”按钮，不能切换为隐私模式。
- 用户点击“新建”产生的空白占位会话仍允许进入隐私模式；进入后立即删除该占位会话，避免历史列表保留空白 item。
- 隐私模式下点击“新建”会自动退出隐私模式，并创建新的普通空白会话。
- 隐私模式下点击“保存”会持久化当前隐私对话、退出隐私模式，并将该对话加入历史列表。
- 隐私模式保存成功后，如果存在可用的标题生成模型，则使用当前隐私会话的完整聊天记录重新生成一次会话标题；标题生成失败时保留保存时的默认标题。
- 隐私模式已有消息时，点击历史会话需要通过符合当前主题的确认弹窗确认；用户取消时继续保留当前隐私对话，避免静默丢失未保存内容。
- 隐私会话保存为普通历史会话时，ID 从 `private-session-*` 规范化为 `session-*`，避免持久化数据继续带隐私态前缀。
- 隐私模式只影响聊天会话持久化，不改变模型请求、上下文拼接、附件校验和发送快捷键等既有行为。

## 状态模型

在 `useAppStore` 中新增运行期状态：

- `privateModeActive`：当前是否处于隐私模式。
- `privateChatSession`：仅内存存在的隐私会话。

新增动作：

- `enterPrivateMode()`：进入隐私模式；必要时删除当前空白占位会话。
- `savePrivateChatSession()`：将隐私会话持久化为普通会话并退出隐私模式。

普通历史会话仍使用 `chatSessions` 和 `activeSessionId`。隐私会话不进入 `chatSessions`，避免历史列表、同步快照和导出数据误包含隐私内容。

## 数据流

`ChatPanel` 显示的 active session 需要优先读取 `privateChatSession`，否则读取 `chatSessions` 中的 `activeSessionId`。

发送消息时，如果 `privateModeActive` 为 true：

1. 构造用户消息和请求消息保持不变。
2. 更新 `privateChatSession.messages`，不调用 `saveChatSession`。
3. 流式回复的 chunk、thinking、complete 都只更新内存隐私会话。
4. 非流式回复完成后只更新内存隐私会话。
5. 不触发基于 IndexedDB 的标题生成流程。

普通模式保留原有持久化流程。

保存隐私会话时先写入普通历史会话，再把完整聊天消息按“用户/助手”顺序整理为标题生成上下文，请求标题模型生成历史列表标题。标题生成是保存后的增强步骤，失败时不回滚已保存会话。

历史会话切换由 UI 层弹出主题化确认弹窗，并由 store 层默认拒绝在未显式确认时丢弃已有消息的隐私会话。这样即使未来新增其他历史入口，也不会绕过保护直接清空 `privateChatSession`。

## UI

在 `.chat-export-trigger` 按钮右侧新增隐私按钮：

- 可进入隐私模式时显示“隐私”。
- 激活隐私模式后显示“保存”，并使用主题 `--color-primary` 作为边框激活色。
- 已存在且包含消息的普通会话不显示该按钮。
- 按钮沿用 `ui-button-secondary`，只增加状态类，不引入新视觉体系。

## 测试

新增或更新以下测试：

- Store：隐私模式发送消息不写入 IndexedDB、不新增 `chatSessions` item。
- Store：空白占位会话进入隐私模式时被删除。
- Store：保存隐私会话后写入 IndexedDB、退出隐私模式并进入历史列表。
- Store：保存隐私会话后用完整聊天记录请求标题模型，并将生成标题写回普通历史会话。
- Store：保存隐私会话后 ID 使用普通 `session-*` 前缀。
- Store：直接切换历史会话不会静默丢弃已有消息的隐私对话。
- UI：隐私模式已有消息时点击历史会话弹出主题化确认弹窗，取消后保留隐私对话，且不调用原生 `window.confirm`。
- UI：隐私按钮位于导出按钮右侧，激活后显示“保存”和激活边框类。
- UI：已存在且有消息的历史会话不显示“隐私”按钮。

## 风险

主要风险是聊天发送流程已有流式和非流式两条路径，隐私模式必须同时覆盖，避免某条路径继续调用 `updateChatSession`。通过 store 单元测试覆盖非流式保存边界，并保留构建、类型检查和现有 App 测试作为回归验证。
