# 渠道模型管理设计

## 背景

当前设置页只有一个“模型名称”输入框，无法表达“一个供应商/渠道下配置多个模型”的真实使用方式。后续需要支持类似 NewAPI 的渠道管理：用户先配置渠道，再在渠道下添加多个模型，并在对话输入区选择具体模型。

## 目标

- 支持添加多个渠道。
- 每个渠道支持添加多个模型。
- 符合 Chrome Side Panel 场景，用户拖动插件宽度时布局可响应。
- 对话区模型选择器显示“渠道 / 模型”，避免多个供应商下模型重名时混淆。
- 保持亮色、清晰、工具型界面，不使用普通后台大屏管理台布局。
- 宽面板空间优先用于设置模块导航，而不是把渠道管理拉成宽表格。

## 非目标

- 不在本轮实现 WebDAV/S3。
- 不在本轮实现模型自动拉取列表。
- 不在本轮实现复杂权限、额度、分组路由。
- 不在窄面板中使用横向溢出的宽表格。

## 数据模型

### ModelProvider

渠道保存供应商级配置：

- `id`
- `name`
- `endpointType`
- `endpointUrl`
- `apiKey`
- `enabled`
- `createdAt`
- `updatedAt`

### ProviderModel

模型保存渠道下的模型配置：

- `id`
- `providerId`
- `displayName`
- `modelId`
- `temperature`
- `maxTokens`
- `systemPrompt`
- `isTitleModel`
- `enabled`
- `createdAt`
- `updatedAt`

## 界面设计

### 窄面板

适用于 360-520px 左右宽度：

- 单列堆叠。
- 渠道列表显示在上方。
- 渠道列表标题右侧保留唯一的“新增渠道”按钮。
- 当前渠道详情显示在渠道列表下方。
- 模型列表用卡片展示，每张卡片显示模型显示名、`model_id`、温度、最大 tokens。
- 当前渠道详情底部只保留“添加模型”，不再重复提供“新增渠道”。

### 宽面板

适用于 640px 以上宽度：

- 使用设置级双栏。
- 左侧为 Tab 导航栏，例如“渠道管理”“提取规则”“同步设置”“界面偏好”。
- 右侧为当前 Tab 内容区。
- 渠道管理内容区仍使用窄面板设计：渠道列表在上，当前渠道详情在下，模型列表用卡片展示。
- 渠道管理内容区设置最大宽度，避免在宽面板下被拉成后台表格。
- 宽面板不使用渠道模型表格。

## 交互规则

- 新增渠道后自动选中新渠道。
- 删除渠道时必须同时删除该渠道下模型，删除前给出确认。
- 没有渠道时显示空态，并提供“新增渠道”按钮。
- 没有模型时显示空态，并提供“添加模型”按钮。
- 对话模型选择器只展示启用渠道下的启用模型。
- 模型选项文本格式为：`渠道名称 / 模型显示名`。
- 如果当前选中模型被删除或禁用，则清空当前选择并禁用发送按钮。
- 宽面板点击左侧 Tab 时，只切换右侧设置内容，不改变渠道管理内部的卡片式布局。

## 验收标准

- 设置页可以展示多个渠道。
- 每个渠道可以展示多个模型。
- 窄面板只出现一个渠道级“新增渠道”入口。
- 当前渠道详情底部只出现“添加模型”入口。
- 宽面板左侧显示设置 Tab 导航栏。
- 宽面板渠道管理仍使用卡片式模型列表，不出现模型表格。
- 对话区模型下拉按“渠道 / 模型”展示选项。
- 未配置可用模型时仍提示用户配置 API Key，并禁用发送按钮。
- 单元测试覆盖渠道新增、模型新增、模型选择展示和删除后的选择失效。
- E2E 覆盖亮色响应式设置页的关键布局。

## 风险

- 如果把渠道和模型仍混在一个 `ModelConfig` 里，后续同步、校验和 UI 都会继续变复杂。本轮应拆分为 `ModelProvider` 与 `ProviderModel`。
- Side Panel 宽度变化频繁，不能依赖固定宽度表格；渠道管理在所有宽度下都应保持卡片式布局。
- API Key 属于渠道级敏感信息，同步前仍必须加密。
