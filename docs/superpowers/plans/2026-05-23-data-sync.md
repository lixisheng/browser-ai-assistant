# 数据同步功能 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 实现 Chrome Sync、WebDAV、S3 兼容存储三种数据备份与恢复能力，并支持独立的同步开关、定时自动同步和可选加密。

**Architecture:** 在 `src/shared/sync/` 下新增同步配置、备份快照、远程 provider 和编排服务，Side Panel 通过 store 调用共享服务，background 通过 runtime message 和 `chrome.alarms` 执行定时备份。恢复时全量覆盖 IndexedDB，并重新加载前端状态。

**Tech Stack:** React 19、Zustand、Dexie、Web Crypto、Chrome Extension MV3、`chrome.storage.sync`、`chrome.alarms`、Fetch、Vitest、Testing Library。

**执行约束:** AGENTS.md 禁止未授权提交，本计划不包含 `git commit` 步骤。实施完成后只运行验证命令并汇报结果。

**资料依据:**
- Chrome Storage 官方文档：`chrome.storage.sync` 总量 `102400` 字节、单项 `8192` 字节、写入频率限制。
- Chrome Alarms 官方文档：使用 `alarms` 权限，Chrome 120 后最小周期可到 `0.5` 分钟；本项目 UI 按用户要求使用分钟，计划最小值设为 `1`。
- AWS S3 SigV4 官方文档：S3 REST 请求使用 Authorization header，签名基于 canonical request、string to sign、派生签名密钥。
- RFC 4918：WebDAV 基于 HTTP 扩展，MVP 仅使用基础 `PUT`/`GET` 文件读写能力。

export async function exportAllDataForSync() {
  const [modelConfigs, modelProviders, providerModels, extractionRules, chatSessions, chatFolders, appSettings] =
    await Promise.all([
      db.modelConfigs.toArray(),
      db.modelProviders.toArray(),
      db.providerModels.toArray(),
      db.extractionRules.toArray(),
      db.chatSessions.toArray(),
      db.chatFolders.toArray(),
      db.appSettings.toArray(),
    ]);

  return { modelConfigs, modelProviders, providerModels, extractionRules, chatSessions, chatFolders, appSettings };
}

export async function replaceAllDataFromSync(snapshot: SyncDataSnapshot): Promise<void> {
  await db.transaction(
    "rw",
    [db.modelConfigs, db.modelProviders, db.providerModels, db.extractionRules, db.chatSessions, db.chatFolders, db.appSettings],
    async () => {
      await Promise.all([
        db.modelConfigs.clear(),
        db.modelProviders.clear(),
        db.providerModels.clear(),
        db.extractionRules.clear(),
        db.chatSessions.clear(),
        db.chatFolders.clear(),
        db.appSettings.clear(),
      ]);
      await Promise.all([
        db.modelConfigs.bulkPut(snapshot.modelConfigs),
        db.modelProviders.bulkPut(snapshot.modelProviders),
        db.providerModels.bulkPut(snapshot.providerModels),
        db.extractionRules.bulkPut(snapshot.extractionRules),
        db.chatSessions.bulkPut(snapshot.chatSessions),
        db.chatFolders.bulkPut(snapshot.chatFolders),
        db.appSettings.bulkPut(snapshot.appSettings),
      ]);
    },
  );
}

- [ ] **Step 4: 实现 SigV4**

实现要点：
- 使用 `crypto.subtle.digest("SHA-256", ...)` 计算 hex hash。
- 使用 `crypto.subtle.importKey` + `crypto.subtle.sign("HMAC", ...)` 派生 signing key。
- canonical headers 固定为 `host;x-amz-content-sha256;x-amz-date`。
- canonical URI 使用 URL pathname，并保证每段 encode。
- credential scope 为 `<date>/<region>/s3/aws4_request`。

- 校验 Endpoint URL、Bucket、Access Key、Secret Key。
- `PUT`/`GET` 都走 SigV4。
- 错误消息脱敏，不输出 Secret Key、Authorization header。
