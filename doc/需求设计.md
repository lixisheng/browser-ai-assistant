# Web AI Chat Extension 需求设计文档

## 1. 产品概述

### 1.1 产品愿景

开发一款功能强大、高度定制化的 Chrome 浏览器插件，使浏览器具备“理解”当前网页内容的能力，并能基于此内容与各种主流大语言模型（LLM）进行智能对话。通过灵活的内容提取、多模型支持和多端同步，打造个性化的网页 AI 助手体验。

### 1.2 核心价值

- **上下文感知:** 将网页内容转化为对话背景，实现“就地”提问和解答。
- **高度自由:** 支持接入各类主流 AI 模型（OpenAI, Anthropic, Gemini 等）。
- **精准提取:** 通过 CSS/XPath 精确控制要作为上下文的网页部分，避免无关信息干扰。
- **无缝衔接:** 多端配置和聊天记录同步，保持一致的使用体验。

## 2. 目标用户与使用场景

### 2.1 目标用户

- **研究人员/学生:** 阅读长篇文档、论文时，需快速总结、解释概念。
- **开发者:** 阅读技术文档、代码片段时，需快速理解、寻求示例或纠错。
- **内容创作者:** 浏览网页素材时，进行头脑风暴、改写或翻译。
- **普通用户:** 阅读长文章、新闻时，需要快速获取摘要或针对特定段落提问。

### 2.2 核心场景示例

- 用户在 GitHub 阅读一个复杂的 Issue，通过侧边栏插件基于当前页面内容调用自定义的 Claude 3.5 模型询问解决方案。
- 用户在维基百科阅读一篇外文长文，配置正则匹配维基百科 URL，自动提取正文内容（CSS选择器），并使用自定义的 Gemini 模型进行翻译和总结。
- 用户在办公室的电脑上配置了 API Key 和同步策略，回到家中使用家里的电脑，配置和聊天记录自动同步，可继续之前的对话。

## 3. 功能需求详细说明

### 3.1 网页内容提取与处理

- **全局模式:** 默认读取当前网页 `<html>` 下所有文本元素中的全局可见文本。
- **精准提取模式 (按 URL 规则匹配):**
   - 用户可配置规则列表：`(URL 正则表达式, 提取规则列表)`。
   - `提取规则`可以是 `CSS 选择器` 或 `XPath`。
   - 支持一个规则包含多个选择器/XPath；配置界面使用一个多行文本域保存，一行一个，提取时按行顺序排序和拼接。
   - **处理逻辑:** 当用户打开一个网页时，插件匹配 URL 正则。若匹配成功，则使用对应的 CSS/XPath 提取内容；若 URL 未匹配、CSS/XPath 未命中、表达式执行失败或提取结果为空，则统一回退到全局模式。
- **内容预处理:** 去除提取内容中冗余的 HTML 标签、脚本、样式，仅保留纯文本结构，以减少 token 消耗。
- **超长内容处理:** MVP 默认从开头截取超长网页内容，并在对话输入区域提示用户“内容被截断，请细化 CSS/XPath”。
- **MVP 限制:** MVP 不支持用户选中文本提问，不检测页面内容变化，不支持手动刷新上下文。

### 3.2 大模型集成与管理 (自定义端点)

- 支持用户自定义添加和管理多个 AI 模型配置。
- 支持配置多个渠道、多个模型，形态类似 NewAPI；用户可在对话输入框处随时切换当前使用的模型。
- MVP 每个模型配置包含：模型名称、模型 ID (`model_id`)、API Key、Endpoint URL、请求格式类型、`temperature`、`max_tokens`、系统提示词。
- API Key 保存时进行校验；校验失败时允许用户继续保存，并在界面中明确提示校验失败原因。
- **支持的请求/响应协议格式 (核心难点):**
   - **OpenAI Chat Completions:** `(/v1/chat/completions)` - 标准 OpenAI 格式，MVP 支持，并兼容 OpenAI-compatible 第三方服务。
   - **Anthropic Messages:** `(/v1/messages)` - Claude 系列模型的标准格式，MVP 支持。
   - **OpenAI Responses API:** `(/v1/responses)` - OpenAI 新的 Responses API，MVP 暂不支持。
   - **OpenAI Response Compaction:** `(/v1/responses/compact)` - Responses 紧凑模式，MVP 暂不支持。
   - **Gemini GenerateContent:** `(/v1beta/models/{model}:generateContent)` - Google Gemini 的格式，MVP 暂不支持。
- **请求模式:** MVP 支持在对话输入框处快捷切换非流式/流式请求。
- **会话管理:** 在与 LLM 对话时，需将提取的网页内容作为 `System Prompt` 或对话历史的初始 `User Message`（根据不同模型最佳实践处理）。
- **失败处理:** 模型请求失败时，允许用户重试；失败消息不保存为正式聊天记录，重试时覆盖当前失败状态。

### 3.3 交互界面 (UI/UX)

- **展现形式:** 浏览器**侧边栏 (Side Panel)**。避免遮挡主要网页内容，提供沉浸式的并排阅读/对话体验。(使用 Chrome 的 `chrome.sidePanel` API)。
- **打开方式:** 支持点击插件图标、快捷键、右键菜单打开侧边栏。
- **聊天界面:**
   - 消息气泡展示（区分 User 和 AI）。
   - 支持 Markdown 渲染（包含代码块高亮）。
   - 输入框（支持多行、快捷键发送）。
   - 切换当前对话所用模型的下拉菜单。
   - 首次使用且未配置 API Key 时，在对话输入框处提示用户完成配置。
   - 支持在输入框处切换当前请求的流式/非流式模式。
- **设置界面:**
   - 模型管理（增删改查 API/Endpoint）。
   - 提取规则管理（URL正则、CSS/XPath 配置界面）。
   - 同步设置界面。
   - 设置修改后立即生效。

### 3.4 数据同步功能

- **同步内容:** 插件配置（模型信息、提取规则、UI偏好等）、聊天记录（按域名或会话分组保存）、链接凭据等均需要加密后同步存储。
- **支持的同步策略 (多选一):**
   - **Chrome Sync:** 使用 `chrome.storage.sync`。MVP 支持，但需要处理严格配额限制。
   - **WebDAV:** 用户配置 WebDAV 服务器地址、用户名、密码。插件将数据序列化为 JSON 文件上传/下载，MVP 暂不支持。
   - **S3 兼容存储:** (如 AWS S3, Cloudflare R2, MinIO)。用户配置 Endpoint, Access Key, Secret Key, Bucket Name，MVP 暂不支持。
- **加密要求:** API Key、链接凭据、聊天记录和配置同步前都必须进行对称加密。对称加密密钥不参与同步，仅持久化在本地，由用户自行设置和记忆；提供是否启用加密功能开关，在关闭状态时无需设置“加密秘钥”，开启时则必须设置，默认为关闭状态；用户忘记密钥时无法恢复已加密的同步数据，配置界面必须给出明确警告。
- **密钥修改:** 本地加密密钥允许修改；输入或保存本地密钥本身不触发远端备份，备份仅由用户手动备份或定时自动同步触发。
- **同步逻辑:** MVP 仅支持定时备份、用户手动备份、用户手动恢复；暂不支持多端同时增删改查场景下的合并。
- **冲突策略:** 用户需要配置备份文件前缀，前缀保存在本地，用于区分不同端的备份；一个前缀只保留一个远程备份，恢复时直接使用当前前缀对应备份覆盖本地，不做合并。
- **配额处理:** 当加密后的同步数据超过 `chrome.storage.sync` 配额时，备份失败并提示用户减少历史记录或关闭部分同步内容；MVP 不自动拆分为多份远程备份。

### 3.5 聊天记录管理

- **本地存储:** 默认使用 IndexedDB (`chrome.storage.local` 可能不够存大量历史) 存储本地聊天记录。
- **历史查询:** 侧边栏提供历史会话列表，支持点击恢复对话上下文。
- **记录范围:** 仅保存所有大模型对话数据，例如 `role`、`content`、`createdAt`、`modelId`、`endpointType`、`streamMode`、`systemPrompt`、`contextPrompt` 等字段，不保存 URL、原始 DOM、网页标题等额外网页元数据。
- **标题生成:** 默认按对话产生顺序排序，用户可单独拖拽排序。每次首次会话使用“对话标题总结模型”基于对话数据生成标题；未配置标题总结模型或标题生成失败时，使用 `网站 title + yyyy-MM-dd HH:mm:ss` 命名，时间使用用户本地时区的 24 小时制，并提供手动修改标题入口。
- **导出/清理:** 支持清空记录；MVP 暂不支持导出聊天记录。

## 4. 非功能需求

- **隐私与安全:** API Key 必须加密或妥善保存在本地，绝不能以明文上传到未经用户授权的第三方服务器。所有的网络请求直接由用户浏览器发出。
- **性能要求:** 网页内容提取必须高效，不阻塞主线程。侧边栏加载速度快。
- **跨平台兼容:** 确保在 Windows/macOS/Linux 的 Chrome 或 Chromium 内核浏览器（如 Edge）中正常运行（主要依赖 Manifest V3 的兼容性）。

## 5. 技术架构设计与选型建议

### 5.1 基础框架

- **Manifest V3 (MV3):** Chrome 插件的最新标准，强制要求。
- **前端框架:** React + Vite + TypeScript。
- **状态管理:** Zustand。
- **UI 组件库:** Tailwind CSS + Radix UI。
- **Markdown 与代码高亮:** 使用成熟稳定的 Markdown 渲染库，并结合 Shiki 或 highlight.js 实现代码块高亮。
- **测试框架:** 使用 Vitest 做单元测试，使用 Playwright 做关键流程冒烟验证。

### 5.2 核心模块职责划分

1. **Background Script (Service Worker):**
   - 处理模型 API 的网络请求 (避免跨域 CORS 问题)。
   - 处理 WebDAV/S3 的数据同步逻辑后台任务。
   - 处理 `chrome.sidePanel` 的开关逻辑。
2. **Content Script:**
   - 注入到当前网页。
   - 根据 Background 传来的 URL 规则和 CSS/XPath 提取当前页面 DOM 元素文本。
   - 后续版本可监听网页 URL 变化（针对 SPA 单页应用）并通知侧边栏更新上下文；MVP 暂不支持页面内容变化检测。
3. **Side Panel (前端 UI):**
   - 渲染聊天界面和设置界面。
   - 与 Background Script 通过 `chrome.runtime.sendMessage` 通信。
   - 使用 Zustand 管理本地状态。

### 5.3 存储方案选型

- `chrome.storage.sync`: MVP 用于保存加密后的同步备份；受配额限制，超限时提示用户处理，不自动拆分多份备份。
- `IndexedDB` (使用 `Dexie.js` 封装): 存储本地聊天记录和复杂的提取规则列表。

## 6. 需求优先级与迭代计划建议

### Phase 1: MVP (最小可行性产品) - 核心体验闭环

- 实现侧边栏 UI。
- 支持点击插件图标、快捷键、右键菜单打开侧边栏。
- 实现**全局网页文本提取**。
- 实现 URL 正则 + 多行 CSS/XPath 的精准内容提取配置；未匹配或匹配失败时回退全局可见文本。
- 支持 **OpenAI Chat Completions** 和 **Anthropic Messages** 格式。
- 支持 OpenAI-compatible 第三方服务。
- 支持流式/非流式请求切换。
- 支持本地聊天记录保存 (`IndexedDB`)。
- 支持 Chrome Sync 加密备份和恢复。
- 支持对话标题生成与默认标题兜底。

### Phase 2: 高阶配置与模型扩展

- 增加对 Gemini API 格式的支持。
- 实现 OpenAI Responses 和 Compaction 端点支持。
- 支持用户选中文本提问。
- 支持页面内容变化检测和手动刷新上下文。

### Phase 3: 多端同步与完善

- 接入 WebDAV 同步功能。
- 接入 S3 同步功能。
- 提供聊天记录的管理、导出功能。
- UI 细节打磨和性能优化。
